01 — Overview

What Is OpenClaw?

Most AI coding agents — Claude Code, Pi, Codex — are powerful but ephemeral: they run in a terminal, process a task, and exit. The moment you close the session, memory is lost, channels go dark, and the agent is unreachable from your phone. OpenClaw solves this by acting as a persistent gateway and deployment runtime that keeps your agent alive 24/7, reachable from any messaging platform.

OpenClaw is self-hosted (runs on your hardware, your rules), open source (MIT licensed, 188k+ GitHub stars as of early 2026), and built for agent-native workloads — tool use, session persistence, memory, and multi-platform routing.

The Core Idea

OpenClaw is not the AI model and not the agent framework. It is the infrastructure layer between your agent and the world: normalising incoming messages from WhatsApp, Telegram, Discord, and 6+ more platforms; managing persistent sessions; enforcing security policy; and injecting skills and memory into each turn.

What Problems Does It Solve?

1
Always-On Availability
The Gateway process runs as a daemon on your machine or a server. Send a WhatsApp message from your phone at midnight and get a response — no terminal needed.
2
Channel Normalisation
WhatsApp, Telegram, Discord, iMessage, Slack, Signal, and Teams all have different message formats, media types, and auth flows. OpenClaw adapts them all to a unified internal model so your agent code sees the same input regardless of source.
3
Session Isolation
Each conversation has its own isolated session store — history, tool state, and memory. A work conversation on Telegram doesn't bleed into a personal conversation on WhatsApp.
4
Persistent Memory
Every turn is written to a JSONL transcript. MEMORY.md files accumulate durable knowledge. When context windows fill, OpenClaw runs compaction — summarising old history — so the agent retains the gist of past conversations indefinitely.
5
Skill Injection
Skills — SKILL.md files with YAML frontmatter — are selectively injected into each turn's system prompt based on relevance signals. The agent gains capabilities without those instructions being present in every token count.
6
Security & Policy Enforcement
A 5-level tool policy chain (global → provider → agent → session → sandbox) governs what the agent can and cannot do. Allowlists restrict which senders can reach the agent. Approval gates halt dangerous actions for human review.
02 — Core Architecture

The Gateway Architecture

OpenClaw's central component is the Gateway — a single long-lived Node.js process that owns all messaging surfaces, exposes a WebSocket API and browser Control UI, and emits system events like cron and heartbeat. Everything flows through it.

Fig 1 — OpenClaw Gateway: channels in, isolated agents out

Key Architectural Properties

🔄

Single Daemon Process

One Node.js Gateway process handles all channels simultaneously. Installed as a system daemon (openclaw onboard --install-daemon) so it survives reboots and restarts automatically.

🌐

Control UI

A browser dashboard at http://127.0.0.1:18789/ provides chat, session management, configuration editing, and live node status. Accessible remotely via Tailscale or SSH tunnel.

📱

Mobile Nodes

iOS and Android nodes extend OpenClaw with Canvas, camera/screen capture, and device actions — enabling workflows that require visual context from a phone.

🔌

Plugin Architecture

Additional channels (Mattermost, Teams, custom integrations) are added as extension packages installed alongside the core Gateway, keeping the core lean.

03 — Messaging Channels

Channel Adapters

OpenClaw's channel adapters normalise the wildly different APIs of each messaging platform into a single internal message format. Each adapter handles authentication, media transcoding, group/DM routing, and reconnection logic — your agent code sees a clean, uniform interface.

Channel Setup Complexity

Each channel requires its own authentication flow: WhatsApp uses QR code pairing, Telegram and Discord use bot tokens, iMessage uses a macOS node. The OpenClaw onboarding wizard (openclaw onboard) guides you through each one interactively.

Channel Auth Method DM Support Group Support Media

Group Chat Controls

In group chats, OpenClaw can be configured to respond only when explicitly mentioned. The requireMention flag and custom mention patterns (e.g. @openclaw) prevent the agent from interrupting every conversation in a busy group.

~/.openclaw/openclaw.json
{ "channels": { "whatsapp": { "allowFrom": ["+15555550123", "+447700900456"], "groups": { "*": { "requireMention": true } } } }, "messages": { "groupChat": { "mentionPatterns": ["@openclaw"] } } }
04 — Agents & Skills

Agents & the Skill System

In OpenClaw terminology, an agent is an isolated AI persona with its own workspace, configuration files, session store, and auth credentials. The Gateway can host one or many agents simultaneously, each completely walled off from the others.

The Agent Workspace

Each agent has a dedicated workspace directory (default ~/.openclaw/workspace) and a private state directory under ~/.openclaw/agents/<agentId>/. The key files that define an agent's identity and behaviour are:

📄

AGENTS.md

The agent's system instructions — its persona, capabilities, and behavioural rules. Injected into every turn's context. The primary prompt engineering surface.

💠

SOUL.md

An optional deeper personality layer — values, tone, communication style. Kept separate from operational instructions so each can evolve independently.

👤

USER.md

Persistent knowledge about the user — preferences, context, working style. Updated over time as the agent learns. Injected as background context each turn.

🧠

MEMORY.md / memory/

Durable knowledge distilled from past conversations. The agent writes lasting notes to dated memory files, which are merged back into future contexts.

Skills

Skills are modular capability extensions — each is a SKILL.md file with a YAML frontmatter block that describes when to activate, and a body that contains the instructions, tool definitions, or workflow templates the agent uses when that skill fires.

Skills live in the workspace's skills/ folder (per-agent) or in ~/.openclaw/skills/ (shared across all agents). They are selectively injected per-turn based on relevance — keeping system prompts lean for most turns while unlocking rich capabilities on demand.

ClawHub — Community Skills Registry

OpenClaw's public registry, ClawHub, hosts over 13,700 community-built skills as of early 2026 — everything from Google Ads management to task queue integration to multi-agent team orchestration. Skills can be installed by pasting a skill's GitHub URL directly into an agent chat session; the agent handles installation automatically.

Example SKILL.md frontmatter
--- name: github-pr-review description: Review and comment on GitHub pull requests triggers: - pull request - PR review - code review tools: - gh version: 1.2.0 --- # GitHub PR Review Skill When the user asks to review a PR, use the `gh` CLI to: 1. Fetch the diff with `gh pr diff <number>` 2. Read existing comments with `gh pr view <number> --comments` ...
05 — Memory & Sessions

Memory, Sessions & Context Management

One of OpenClaw's core differentiators is that it treats memory as a first-class concern rather than delegating it entirely to the model's context window. A multi-layered memory architecture ensures the agent is never starting from scratch.

Fig 2 — OpenClaw memory layers: volatile context at top, permanent identity at bottom

Context Window Management & Compaction

Long-running agents inevitably approach context window limits. OpenClaw's Context Window Guard monitors token counts and triggers a compaction pass before the window overflows. During compaction, the agent is prompted to:

1
Write lasting notes to memory
The agent identifies key facts, decisions, and context from the current session and writes them to a dated memory file (e.g. memory/2026-03-04.md).
2
Summarise old tool outputs
Large tool responses (file reads, API results) that were appended to context are replaced with compact summaries, freeing up tokens without losing semantic content.
3
Resume with a clean window
Conversation continues from the compacted state. The agent retains full long-term memory via Layer 3 files while gaining a fresh context window.

Session Isolation

Each agentId gets a dedicated session store under ~/.openclaw/agents/<agentId>/sessions/. Sessions are keyed by sender identity (e.g. phone number, Discord user ID). Never reuse agentDir across agents — this causes auth and session collisions. If you want to share credentials, explicitly copy auth-profiles.json to the target agent's directory.

06 — Security & Policy

Security Model

OpenClaw's security model is layered and explicit. Rather than trusting the model to stay within safe boundaries by prompt alone, it enforces policy at the infrastructure level — before the agent even sees a request, and after it produces an action.

The 5-Level Tool Policy Chain

Every tool invocation is evaluated against a chain of policy definitions, from broadest to most specific. Each level can permit, deny, or require approval. The most restrictive applicable rule wins.

Fig 3 — 5-level tool policy chain evaluated on every tool invocation

Allowlists & Access Control

The allowFrom config restricts which sender identities can reach each channel. For example, a WhatsApp agent can be locked to a specific phone number or list of numbers — preventing anyone who discovers the bot number from interacting with it.

Lane Queues

Within a session, requests are serialised through a lane queue by default. This prevents race conditions when a user sends multiple messages in rapid succession — each is processed in order, with the previous tool chain fully complete before the next begins. Parallelism is an opt-in configuration for workloads explicitly designed to be safe.

Approval Gates

High-risk actions — file deletions, API calls with side effects, financial operations — can be flagged to require explicit human approval before execution. The agent pauses, surfaces the pending action in the Control UI, and waits for a confirm/deny response. This is the production equivalent of human-in-the-loop guardrails.

Self-Hosted Means Your Responsibility

Because OpenClaw runs on your hardware and uses your API keys, you are responsible for securing the Gateway host, managing model API keys, and configuring appropriate allowlists. The default configuration trusts all local connections — tighten this before exposing the Gateway to the internet or a remote access tunnel.

07 — Multi-Agent Routing

Multi-Agent Routing

OpenClaw's "multi-agent" capability is best understood as multi-persona routing: running multiple isolated agent brains behind different identities or channels. A "work" agent on Telegram and a "personal" agent on WhatsApp coexist within the same Gateway but remain fully isolated — different workspaces, different sessions, different credentials.

Important Distinction

This is fundamentally different from the orchestration patterns in frameworks like LangGraph or the OpenAI Agents SDK, where multiple agents collaborate on a shared task via handoffs and tool calls. OpenClaw agents do not talk to each other by default — they are isolated personas, not collaborating nodes in a graph. Collaborative multi-agent orchestration happens inside one agent's tool use, not between OpenClaw agents.

Per-Agent Isolation

Each agent in the Gateway gets its own completely separate:

📁

Workspace

Separate working directory with its own AGENTS.md, SOUL.md, USER.md, and skills/ folder. No shared file system by default.

🗂️

Session Store

Chat history, routing state, and tool results stored under a dedicated agentId path. Auth profiles are never shared automatically.

🔑

Auth Profiles

Each agent maintains its own auth-profiles.json. To share credentials between agents, explicitly copy the file — no implicit credential sharing.

📣

Channel Bindings

Channels are bound to specific agentIds. Multiple agents can share a channel (e.g. WhatsApp) by routing on sender identity, but each responds from its own brain.

Enabling Agent-to-Agent Communication

For true collaboration — one OpenClaw agent delegating work to another — the community has built skills like agent-team-orchestration and the agentToAgent tool. These enable patterns like: a planner agent dispatching tasks to specialist agents (coder, writer, reviewer) via a shared message queue or channel binding, with results flowing back to the orchestrator. This is a skill-level capability, not a built-in Gateway primitive.

agents list in openclaw.json
{ "agents": { "list": [ { "id": "work", "workspace": "~/.openclaw/workspaces/work", "agentDir": "~/.openclaw/agents/work" }, { "id": "personal", "workspace": "~/.openclaw/workspaces/personal", "agentDir": "~/.openclaw/agents/personal" } ] }, "channels": { "telegram": { "accounts": { "work-bot": { "token": "..." } } }, "whatsapp": { /* personal agent */ } } }
08 — Configuration Reference

Configuration

All Gateway configuration lives in a single JSON file at ~/.openclaw/openclaw.json (or the path set by OPENCLAW_CONFIG_PATH). The file is hot-reloaded — most changes take effect without a Gateway restart.

Start Minimal

If you do nothing, OpenClaw uses its bundled Pi binary in RPC mode with per-sender sessions. The onboarding wizard generates a valid baseline config. Start from that and add complexity only when you need it.

Key Configuration Sections

Section What It Controls Key Fields
agents.defaults Default model, context pruning, compaction policy for all agents model.primary, contextPruning, compaction
agents.list Per-agent workspace paths, agentId, overrides id, workspace, agentDir
channels.* Channel-specific auth, allowlists, group rules allowFrom, requireMention, accounts
gateway.auth Token auth for the Control UI and WebSocket API token
skills.install.nodeManager Package manager used for skill installation npm / pnpm
update.channel OpenClaw update channel stable / beta

Context Pruning Configuration

The contextPruning block under agents.defaults controls how OpenClaw manages context window pressure. Key fields:

"contextPruning": { "mode": "cache-ttl", // prune by age "ttl": "45m", // keep last 45 minutes "keepLastAssistants": 2, // always keep 2 most recent assistant turns "softTrim": { "maxChars": 2500, // trim tool outputs over 2500 chars "headChars": 900, // keep first 900 chars "tailChars": 900 // keep last 900 chars }, "hardClear": { "enabled": true, "placeholder": "[Old tool output cleared]" } }
09 — Quick Start

Getting Started

OpenClaw requires Node.js 22+ and an API key from your chosen LLM provider. For best quality, use the latest frontier model available from your preferred provider.

1
Install OpenClaw globally
npm install -g openclaw@latest — installs the CLI and Gateway process.
2
Run the onboarding wizard
openclaw onboard --install-daemon — interactive guided setup that configures your first agent, sets an API key, and installs the Gateway as a system service.
3
Pair a channel
openclaw channels login — follow the prompts to pair WhatsApp (QR code), Telegram (bot token), Discord, or another platform.
4
Start the Gateway
openclaw gateway --port 18789 — starts the Gateway. If installed as a daemon, it starts automatically at boot. Access the Control UI at http://127.0.0.1:18789/.
5
Install your first skills
Browse awesome-openclaw-skills on GitHub. Paste a skill's GitHub URL into your agent chat — it installs in the background.
6
Tighten security
Edit ~/.openclaw/openclaw.json to add allowFrom allowlists for each channel, enable gateway.auth.token for the Control UI, and review tool policy if exposing to the internet.
